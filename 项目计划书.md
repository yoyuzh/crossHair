# C# 外置准星软件开发计划书

> 调研更新日期：2026-05-20  
> 定位修正：本项目只做 Windows 屏幕中心标记工具，不做游戏数据读取、图像识别、输入自动化、注入或反作弊绕过。公开发布时应明确提示用户：不同游戏、赛事和反作弊系统对外置准星的规则不同，使用前需要确认对应游戏条款。

## 1. 项目定位

项目名称可以暂定为：

```text
CrossMark / CenterDot / AimMarker
```

核心定位：

> 一个 Windows 平台的轻量级屏幕中心标记工具，通过透明置顶窗口在游戏画面上显示自定义准星。

注意：产品定位上不要写成“游戏作弊辅助”，更稳的说法是：

```text
屏幕中心标记工具
准星样式可视化工具
显示器中心辅助标记
Screen Center Marker
```

---

# 2. 调研结论和技术路线

## 2.1 竞品和实现路线结论

| 路线 | 典型产品 / 形态 | 实现原理 | 优点 | 缺点 | 本项目建议 |
| --- | --- | --- | --- | --- | --- |
| 普通桌面透明窗口 | 轻量准星工具、开源 overlay 小工具 | 创建无边框、透明、置顶、鼠标穿透窗口，在窗口中绘制准星 | 开发快、实现透明、风险边界清晰、不需要注入游戏 | 独占全屏基本不可靠，可能被部分反作弊或游戏窗口挡住 | 第一版采用 |
| Xbox Game Bar Widget | Crosshair X、CenterPoint 等 | 通过 Game Bar 小组件覆盖在游戏之上 | 更适合游戏场景，竞品用它解决全屏覆盖问题 | SDK、打包、上架、审核、用户启用流程更复杂 | 第二阶段调研，不进入 MVP |
| 游戏/显卡/显示器自带准星 | 显示器 OSD、驱动 overlay | 显示器硬件或驱动层叠加 | 兼容性最好，不接触游戏进程 | 软件可控性弱，功能边界受硬件限制 | 作为竞品，不作为实现路线 |
| DirectX/OpenGL/Vulkan Hook | 一些游戏 overlay / 作弊工具 | 注入或 hook 图形 API 后在游戏帧内绘制 | 独占全屏兼容强 | 高反作弊风险，技术和合规风险都高 | 明确禁止 |

结论：

```text
MVP 用 WPF + Win32 做普通透明 overlay。
不要为了覆盖独占全屏去做注入、hook 或驱动层方案。
如果后续必须支持更多全屏游戏，优先研究 Xbox Game Bar Widget，而不是 DirectX Hook。
```

## 2.2 第一版技术选型

推荐：

```text
C# + WPF + Win32 API
```

原因：

| 项目     | 选择                                             |
| ------ | ---------------------------------------------- |
| UI 框架  | WPF                                            |
| 系统 API | Win32 API                                      |
| 配置存储   | JSON                                           |
| 快捷键    | RegisterHotKey                                 |
| 托盘图标   | Hardcodet.NotifyIcon.Wpf 或 WinForms NotifyIcon |
| 绘制方式   | WPF Canvas / DrawingVisual                     |
| 打包方式   | MSIX 或 exe installer                           |

---

## 2.3 为什么不用 WinUI 3

WinUI 3 比 WPF 新，但对这种“透明、置顶、鼠标穿透、无边框 overlay”的场景，WPF + Win32 API 更直接。

第一版目标是：

```text
快做出来
稳定覆盖
方便调试
后面能重构
```

所以不建议一开始追求太新框架。

## 2.4 为什么第一版不做 Game Bar

Game Bar 更接近 Crosshair X 这类竞品在游戏内覆盖的实现方向，但第一版先不做，原因是：

```text
1. MVP 的核心风险是准星绘制、DPI、多显示器、配置和稳定性，不是上架流程。
2. Game Bar Widget 会引入额外 SDK、打包、账号、分发和审核成本。
3. 普通 WPF overlay 可以先验证用户体验和样式系统。
4. 如果后续确认独占全屏支持是核心卖点，再单独做 Game Bar 版本。
```

# 3. 核心功能规划

## 3.1 MVP 版本功能

第一版只做最核心功能：

| 功能     | 说明               |
| ------ | ---------------- |
| 屏幕中心准星 | 在主显示器中心绘制准星      |
| 鼠标穿透   | 不影响游戏点击          |
| 透明背景   | 只显示准星，不遮挡画面      |
| 置顶显示   | 覆盖在普通窗口 / 无边框游戏上 |
| 快捷键开关  | 例如 `F8` 显示 / 隐藏  |
| 设置窗口   | 修改颜色、大小、透明度、线宽   |
| 本地配置保存 | 重启后保留上次设置        |
| 托盘图标   | 后台运行、退出、打开设置     |
| 合规提示   | 首次启动展示使用边界和风险提示 |

MVP 应避免做任何“自动识别游戏进程”的逻辑。只要软件主动扫描进程、读取窗口标题、按游戏切配置，就更容易被用户和反作弊系统误解为游戏辅助工具。第一版只允许用户手动选择显示器、样式和开关状态。

# 5. Overlay 实现方案

## 5.1 Overlay 窗口特性

窗口应该具备这些属性：

```text
无边框
透明背景
全屏覆盖
置顶
不显示在任务栏
鼠标穿透
不抢焦点
```

WPF 侧大概是：

```xml
<Window
    WindowStyle="None"
    AllowsTransparency="True"
    Background="Transparent"
    Topmost="True"
    ShowInTaskbar="False"
    ResizeMode="NoResize">
</Window>
```

Win32 扩展样式需要加：

```text
WS_EX_LAYERED
WS_EX_TRANSPARENT
WS_EX_NOACTIVATE
WS_EX_TOOLWINDOW
```

含义：

| 样式                  | 作用             |
| ------------------- | -------------- |
| `WS_EX_LAYERED`     | 分层透明窗口         |
| `WS_EX_TRANSPARENT` | 鼠标穿透           |
| `WS_EX_NOACTIVATE`  | 不抢焦点           |
| `WS_EX_TOOLWINDOW`  | 不显示在 Alt+Tab 中 |

注意：微软文档中 `WS_EX_TRANSPARENT` 的核心语义是让窗口在同线程其他窗口绘制之后再绘制，很多 overlay 示例会用它实现点击穿透，但实际点击穿透建议同时做两层兜底：

```text
1. WPF 层设置 IsHitTestVisible=false。
2. Win32 层处理 WM_NCHITTEST，返回 HTTRANSPARENT。
```

这样可以降低不同 Windows 版本、DPI 和窗口组合下鼠标事件被 overlay 拦截的概率。

---

## 5.2 显示兼容性

需要明确支持范围：

| 游戏模式     | 兼容性     |
| -------- | ------- |
| 窗口模式     | 高       |
| 无边框窗口化   | 高       |
| 独占全屏     | 低 / 不稳定 |
| 反作弊强管控游戏 | 不保证     |

第一版产品说明应写：

```text
推荐在无边框窗口化模式下使用。部分独占全屏游戏可能无法显示。
本工具不保证兼容所有游戏和反作弊环境。
```

## 5.3 Overlay 边界和安全约束

为了降低被反作弊误判和被用户误用的风险，第一版实现必须遵守：

```text
不读取游戏进程
不枚举游戏内对象
不读取显存或帧缓冲
不截图识别敌人或 UI
不发送键鼠输入
不使用低级键鼠钩子做宏
不 DLL 注入
不 DirectX / Vulkan / OpenGL Hook
不修改游戏文件
不隐藏进程
不规避反作弊检测
```

允许做：

```text
创建普通 Windows 顶层透明窗口
在自己的窗口里绘制静态准星
用 RegisterHotKey 注册少量全局快捷键
读取和保存自己的本地配置文件
显示托盘菜单和设置窗口
```

# 6. 准星样式设计

## 6.1 第一版样式

建议先做 4 种：

```text
Dot        圆点
Cross      十字
Circle     圆圈
CrossDot   十字 + 中心点
```

配置字段：

```csharp
public class CrosshairSettings
{
    public bool Enabled { get; set; } = true;
    public CrosshairStyle Style { get; set; } = CrosshairStyle.CrossDot;

    public double Size { get; set; } = 16;
    public double Thickness { get; set; } = 2;
    public double Gap { get; set; } = 4;
    public double Opacity { get; set; } = 0.9;

    public string Color { get; set; } = "#00FF00";

    public double OffsetX { get; set; } = 0;
    public double OffsetY { get; set; } = 0;

    public string MonitorId { get; set; } = "Primary";
    public bool ComplianceNoticeAccepted { get; set; } = false;
    public string ComplianceNoticeVersion { get; set; } = "2026-05-20";
}
```

---

## 6.2 绘制逻辑

中心点：

```text
centerX = screenWidth / 2 + offsetX
centerY = screenHeight / 2 + offsetY
```

十字准星可以分成 4 条线：

```text
左线：centerX - gap - size 到 centerX - gap
右线：centerX + gap 到 centerX + gap + size
上线：centerY - gap - size 到 centerY - gap
下线：centerY + gap 到 centerY + gap + size
```

圆点：

```text
以中心为圆心，绘制半径 dotRadius 的圆
```

圆圈：

```text
以中心为圆心，绘制半径 circleRadius 的空心圆
```

# 8. 快捷键设计

## 8.1 第一版快捷键

第一版默认只注册一个全局快捷键：

| 快捷键  | 功能        |
| ----- | --------- |
| `F8` | 显示 / 隐藏准星 |

其他快捷键不要默认注册，可以在后续版本让用户手动开启：

| 快捷键              | 功能     |
| ---------------- | ------ |
| `F9`             | 打开设置窗口 |
| `Ctrl + Alt + R` | 重置准星位置 |
| `Ctrl + Alt + Q` | 退出程序   |

---

## 8.2 注意事项

不要默认占用太多按键，尤其是：

```text
Shift
Ctrl
Alt
鼠标侧键
空格
左键
右键
数字键
```

这些很容易和游戏冲突。

不要实现压枪、连点、自动开火、自动切枪等任何输入自动化。即使用户要求，也应作为产品红线拒绝。

# 9. 开发阶段计划

## 阶段一：MVP 原型

目标：能显示一个稳定的准星。

任务：

```text
1. 创建 WPF 项目
2. 创建 OverlayWindow
3. 设置透明、置顶、鼠标穿透
4. 在屏幕中心绘制十字准星
5. 支持 F8 显示 / 隐藏
6. 支持托盘退出
7. 首次启动展示合规提示
```

验收标准：

```text
打开程序后，屏幕中心显示准星
鼠标点击不会被准星窗口拦截
按 F8 可以隐藏 / 显示
在无边框窗口化游戏中可见
首次启动必须确认合规提示
不包含进程扫描、注入、hook、截图识别或输入自动化代码
```

---

## 阶段二：设置系统

目标：用户可以自定义准星。

任务：

```text
1. 添加设置窗口
2. 添加颜色选择
3. 添加大小、线宽、间距、透明度滑块
4. 实时更新 Overlay
5. 保存 JSON 配置
6. 启动时读取配置
```

验收标准：

```text
修改设置后准星立即变化
关闭再打开后配置仍然存在
设置窗口不影响 Overlay 运行
```

---

## 阶段三：多显示器和 DPI

目标：减少不同设备上的偏移问题。

任务：

```text
1. 获取所有显示器
2. 支持选择显示器
3. 监听显示器变化
4. 处理 Windows DPI 缩放
5. 分辨率变化后自动重算中心点
```

验收标准：

```text
125%、150%、200% 缩放下准星不偏
外接显示器可选择
分辨率变化后准星仍在中心
```

---

## 阶段三补充：全屏覆盖路线评估

目标：判断是否需要开发 Game Bar Widget 版本。

任务：

```text
1. 调研 Xbox Game Bar Widget SDK 的项目模板、打包和分发限制
2. 验证 widget 是否能满足准星静态绘制、点击穿透和设置交互
3. 比较普通 WPF overlay 与 Game Bar 版本的兼容性
4. 明确是否值得维护两个前端外壳
```

验收标准：

```text
有一份独立评估文档
能说明 Game Bar 版本解决了什么问题
能说明它增加了哪些发布和维护成本
没有引入 DirectX Hook 或注入方案
```

---

# 10. 风险评估

## 10.1 技术风险

| 风险      | 说明                    | 处理              |
| ------- | --------------------- | --------------- |
| 独占全屏不可用 | Windows 普通置顶窗口压不过独占全屏 | 要求用户使用无边框窗口化    |
| 反作弊拦截   | 某些游戏会限制 overlay       | 不注入、不 hook、不读内存 |
| DPI 偏移  | 高分屏下中心点可能不准           | 做 DPI-aware     |
| 多显示器问题  | 准星显示到错误屏幕             | 添加显示器选择         |
| 快捷键冲突   | 游戏可能占用 F8/F9          | 允许用户自定义         |

---

## 10.2 合规风险

不同游戏和反作弊系统的规则不完全一致。外置准星本身不等于自动作弊，但在某些游戏、赛事或平台规则中，任何提供额外视觉辅助的第三方工具都可能被禁止。产品文案和实现都要保持保守。

软件必须避免：

```text
读取游戏内存
识别敌人
自动瞄准
自动开火
压枪宏
DLL 注入
DirectX Hook
绕过反作弊
修改游戏文件
```

你只做：

```text
透明窗口显示一个中心标记
```

风险会低很多。

## 10.3 合规设计原则

| 原则 | 具体要求 |
| --- | --- |
| 只显示自己的窗口 | 不读、不写、不 hook 游戏进程 |
| 不提供信息优势 | 不识别敌人、血量、轮廓、后坐力、弹道或游戏 UI |
| 不自动输入 | 不实现宏、压枪、连点、自动开火、热键连招 |
| 不规避检测 | 不隐藏进程、不混淆目的、不绕过反作弊 |
| 文案保守 | 不承诺“防封”“反检测”“所有游戏可用” |
| 用户知情 | 首次启动和 README 提醒用户遵守对应游戏规则 |
| 可审计 | 核心 overlay 代码保持简单，便于开源或接受用户检查 |

## 10.4 重点游戏 / 平台注意事项

| 场景 | 风险判断 | 产品处理 |
| --- | --- | --- |
| 单机游戏、桌面标记、训练场景 | 风险低 | 正常支持 |
| 无边框窗口化网游 | 中等，取决于游戏条款 | 只声明技术兼容，不声明合规允许 |
| 独占全屏游戏 | 技术兼容差 | 建议切无边框窗口化；不做 hook |
| 强反作弊竞技游戏 | 合规和误判风险较高 | README 单独提示，使用前查游戏规则 |
| 赛事 / 天梯 / 排位 | 风险高 | 不宣传为竞技优势工具 |

公开说明中建议使用：

```text
This app is a screen marker only. It does not read, modify, or control any game.
Some games, tournaments, or anti-cheat systems may restrict third-party overlays.
Please check the rules of the game you play before using it.
```

避免使用：

```text
undetectable
anti-ban
safe for all games
works with any anti-cheat
no recoil
aim assist
```

## 10.5 竞品启发

| 竞品类型 | 可借鉴点 | 不应照搬的点 |
| --- | --- | --- |
| Crosshair X / CenterPoint 类工具 | 样式编辑、分享码、Game Bar 路线、全屏覆盖卖点 | 早期就做复杂商店、云同步和皮肤生态 |

# 12. 最小可行版本范围

第一版不要做太多，建议只做：

```text
透明置顶准星
4 种样式
颜色 / 大小 / 透明度设置
F8 开关
托盘退出
JSON 保存配置
首次启动合规提示
```

不要一开始做：

```text
Game Bar
账号系统
云同步
插件系统
皮肤市场
游戏自动识别
进程检测
DirectX Hook
截图识别
宏和输入自动化
反作弊规避
```

这些会把项目复杂度拉高。

# 15. 判断项目是否成功

第一版成功标准不是“所有游戏全屏都能用”，而是：

```text
在 Windows 桌面和无边框窗口化游戏中稳定显示
不影响鼠标键盘输入
不读游戏数据
资源占用低
用户能快速调整准星
用户清楚知道兼容性和合规边界
```

这才是 C# 外置准星小工具最现实的目标。

# 16. 测试矩阵

## 16.1 技术测试

| 测试项 | 验收标准 |
| --- | --- |
| 桌面点击穿透 | overlay 开启时能正常点击桌面、浏览器和游戏窗口 |
| 窗口模式游戏 | 准星可见，不抢焦点 |
| 无边框窗口化游戏 | 准星可见，不影响鼠标键盘 |
| 独占全屏游戏 | 可以不可见，但软件不能崩溃 |
| DPI 100% / 125% / 150% / 200% | 中心位置不明显偏移 |
| 多显示器 | 能选择目标显示器，拔插显示器后不崩溃 |
| 分辨率变化 | 重新计算窗口尺寸和中心点 |
| 休眠 / 唤醒 | overlay 和托盘仍可正常工作 |
| 配置损坏 | 自动回退默认配置并写日志 |
| 单实例 | 重复启动只唤起已有设置窗口 |

## 16.2 合规自查

每次发布前检查：

```text
代码中没有 OpenProcess 读取游戏进程
代码中没有 ReadProcessMemory / WriteProcessMemory
代码中没有 SetWindowsHookEx 低级键鼠宏逻辑
代码中没有 DirectX / Vulkan / OpenGL Hook
代码中没有 DLL 注入或驱动
代码中没有截图识别、目标检测或自动输入
README 没有防封、反检测、全游戏可用等承诺
首次启动合规提示仍然存在
```

## 16.3 公开发布材料

README 至少包含：

```text
项目是什么：屏幕中心标记工具
项目不是什么：不是自动瞄准、不是压枪宏、不是反作弊绕过工具
兼容性：推荐窗口化或无边框窗口化，独占全屏不保证显示
合规提示：使用前确认游戏、平台和赛事规则
隐私说明：不读取游戏进程，不上传用户数据
卸载说明：配置文件位置和清理方式
```
